Source code files are compiled into object files by the compiler. Object code files are linked into an executable, by the linker.

Symbol Table

An object file contains a symbol table, which maps identifiers (names of functions and variables) used in the program (such as function names, global variable names, etc.) to memory offsets within the object code file. At these memory offsets in the object file is the machine code implementation of the symbol.

For example, for a function named add, the symbol table will contain an entry that maps the text add to a different location in the object file. At that location will be the machine code instructions comprising the body of the add function.

Linker

Identifiers in the symbol table are referred to as symbols.

The linker takes as input a group of object files, resolves all named-references, and produces an executable. A valid linker could essentially copy the machine-code instruction bodies of every function into the executable function, and then replace all the function calls JMP statements.

Libraries

There are two types of libraries:

  1. Static Libraries - simply a bundled group of object files.
  2. Shared (Dynamic) Libraries - a bundled group of object files that were each compiled position-independently. These can be loaded by the operating system's dynamic library loader at runtime.

Object files contain enough information to be able to invoke and execute functions by source-code name. That said, the object file does not store information about the parameter list of a function (i.e. what arguments it should be called with).

Usage

Whether using a static or shared library, a programmer will realistically need the header files that the library was compiled with. Libraries (both static and shared) are always used with their corresponding header files. The headers are included during compilation, and the libraries are referenced during linking.
